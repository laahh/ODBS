01 · Kondisi saat ini

Tantangan data safety
hari ini.

Enam celah operasional yang menghambat transisi menuju Safety Intelligence yang terukur dan dapat dipertanggungjawabkan.

01Data TersebarSumber data masih terpisah di banyak file dan sistem.
02Cleansing Manual BerulangReporting masih memerlukan cleaning dan rutin manual berulang.
03Database Belum TerpusatBelum ada single source of truth untuk data safety.
04Dashboard Belum AnalitisDashboard belum otomatis menunjukkan issue, deviasi, dan prioritas.
05Rekomendasi Masih ManualInsight dan rekomendasi masih sangat bergantung pada evaluator.
06AI Belum DimanfaatkanBelum ada rekomendasi AI berbasis data historis dan pola risiko.
Tanpa fondasi data terpusat, setiap langkah menuju AI berisiko dibangun di atas data yang terfragmentasi dan sulit diaudit.
02 · ODBC hari ini

Fondasi sudah dibangun—
sejak 1 November 2024.

PlatformCATALYST

Central Data Processing sebagai tim dan mesin pengolahan data lintas bisnis.

StorageData Lake · Warehouse · Mart

Lapisan penyimpanan dan analitik existing yang sudah dapat dipakai ulang.

OutputOne Map · Data Services

Konsumsi data dan integrasi lintas fungsi sudah mulai berjalan.

ODBC mengintegrasikan data lintas bisnis melalui Data Management Framework. Kita tidak mulai dari nol—OBDS berdiri di atas fondasi ini.

02 · Kesenjangan

Integrasi luas
belum berarti siap-AI.

ODBC menyatukan data secara luas. AI membutuhkan kedalaman kanonik: entitas inti tunggal, taksonomi ISO 45001, identity resolution, lineage, ownership, serta kualitas data yang terukur.

ODBC StrengthBroad integration

Data lintas bisnis terkumpul dan tersedia untuk kebutuhan analitik umum.

AI RequirementCanonical depth

Definisi tunggal, relasi konsisten, akses terkontrol, dan evidence lineage.

Tanpa lapisan kanonik, LLM dan agent berisiko menghasilkan jawaban meyakinkan di atas data ambigu. OBDS menutup celah tersebut.
03 · ODBC → OBDS

OBDS adalah lapisan domain safety
di atas ODBC.

AI Layer
LLM · Agentic AI · Advanced AnalyticsKonsumen intelligence yang dibangun setelah fondasi safety siap.
OBDS · New
Canonical Model · MDM · Semantic · KnowledgeKedalaman domain keselamatan: entitas inti, taksonomi, golden record, dan data product.
ODBC · Existing
CATALYST · Data Lake · Warehouse · MartDipakai ulang, bukan diganti. OBDS bersifat logical layer di atas infrastruktur existing.
Reuse existing infrastructure → lower cost, lower risk, faster implementation.

09 · Agentic AI

Dari jawaban
ke tindakan terkontrol.

Agent 01Agent Investigasi

Mengumpulkan bukti lintas modul, menyusun kronologi, dan menyiapkan draf laporan.

Agent 02Corrective Action Tracker

Memantau due date, mengingatkan PIC, dan mengeskalasi item overdue.

Agent 03Profil Risiko Kontraktor

Mengkonsolidasikan sinyal CSMS menjadi profil risiko terkalibrasi.

Agent 04Operational Monitoring

Menghubungkan Mining Eyes atau BeSIGMA untuk early alert geofence dan perilaku.

Semua agent menggunakan human-in-the-loop + guardrails. Agent mengusulkan, manusia memutuskan; tidak menggantikan inspeksi dan tidak digunakan otomatis untuk tujuan disipliner.

Dari fragmented data
ke governed intelligence.

L1 · CurrentCatalogedRegistry, owner, steward, dan metadata dasar.
L2 · TargetCanonicalGolden record, taksonomi, DQ, dan data product.

Target jangka menengah adalah L2—Canonical. Kapabilitas LLM dan Agentic AI matang pada L3–L4; tahap fondasi tidak boleh dilompati.
